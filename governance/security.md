## 安全与漏洞披露流程

本项目非常重视安全性。如果你在使用或审计本项目时发现了潜在的安全问题或漏洞，请按照下面的流程进行报告，而不要直接在公开的 Issue 或讨论区中披露细节。

### 1. 报告安全问题

- **报告方式（示例）**：
  - 优先推荐：通过电子邮件发送至项目维护者安全联系人（例如：`g2rain_developer@163.com`）
  - 如没有专门安全邮箱，可使用：项目主要维护者邮箱，或私聊方式（如 GitHub Security Advisories）
- **请在报告中尽量包含**：
  - 漏洞简要描述及影响范围
  - 复现步骤（PoC）或最小复现示例
  - 相关环境信息：版本号、操作系统、配置方式等
  - 如果可能，请给出你建议的修复思路

> 如你不确定某个问题是否属于安全问题，也**欢迎先私下沟通**，由维护者进行判断。

### 2. 响应与沟通

在收到你的安全报告后，我们通常会按照以下流程处理：

1. **确认收悉**  
   - 在3 个工作日内内确认已经收到你的报告；
   - 分配唯一追踪编号（如适用），方便后续沟通。

2. **初步评估**  
   - 评估漏洞严重程度及受影响范围；
   - 视情况请求更多信息或复现细节；
   - 决定是否需要启动紧急修复流程。

3. **制定修复计划**  
   - 制定修复方案与时间表（包括补丁开发、测试与发布计划）；
   - 如需要与相关依赖或下游使用方协调，将在不暴露细节的前提下进行沟通。

4. **持续沟通**  
   - 在整个修复周期内，与报告者保持私下沟通；
   - 在不增加风险的前提下，及时同步进展。

### 3. 修复与披露策略

- **修复优先级**  
  - 高危 / 远程可利用漏洞将被优先修复并尽快发布补丁版本；
  - 中低危问题会在常规迭代中修复。

- **发布时间窗口（示例）**  
  - 视漏洞严重程度和修复复杂度，一般会在 **7～90 天** 的时间窗口内完成修复与披露；
  - 如漏洞极其严重且易于大规模利用，我们可能会缩短窗口或采用分阶段发布策略。

- **安全公告与版本发布**  
  - 在修复发布后，会通过以下至少一种方式披露安全公告：
    - 发布 Release Note / Change Log；
    - 在 `SECURITY.md` / `security.md` 或文档中补充说明；
    - 在 Issue / 讨论区发布公告（不包含可被直接利用的技术细节）。
  - 如果你愿意，我们可以在公告中感谢你的贡献（也可以选择保持匿名）。

### 4. 负责任的披露原则

我们鼓励 **负责任的漏洞披露**：

- 在漏洞尚未修复前，请不要在公开渠道披露完整细节或可直接利用的 PoC；
- 维护者会尽量在合理时间内修复并发布补丁；
- 修复发布后，可以在充分告知用户升级的前提下，逐步公开更多技术细节，用于社区学习和改进安全实践。

### 5. 非安全性问题

对于以下类型的问题，请优先采用**公开 Issue** 的方式反馈，而不是通过安全渠道：

- 功能缺陷、性能问题；
- 文档错误或使用疑问；
- 一般性的设计建议或改进意见；
- 与安全无关的报错日志与异常。

### 6. 致谢

感谢所有帮助提升本项目安全性的贡献者。  
安全是一个持续演进的过程，我们十分欢迎任何负责任的报告与建议。
