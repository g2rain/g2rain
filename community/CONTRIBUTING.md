# 贡献指南（Contributing Guide）

感谢你对 **g2rain** 的关注与贡献意愿！  
为了让协作更顺畅、更高效，请在参与项目前阅读本指南。

---

## 1. 我可以如何贡献？

无论你是刚接触 g2rain，还是已经深入使用，都有很多方式可以参与：

- **使用反馈**：在实际项目中使用 g2rain，并反馈遇到的问题与需求。
- **文档改进**：修正文档中的错别字、过期内容、示例不清晰等问题。
- **Bug 修复**：修复现有 Issue 中标记的 Bug，或你自己发现的问题。
- **新特性**：为平台提出或实现新功能（建议先通过 Issue / RFC 讨论）。
- **示例与教程**：补充 Demo、最佳实践、集成案例等。

---

## 2. 提交 Issue 前请先检查

在创建 Issue 之前，请先：

1. 搜索现有 Issue，避免重复提问。
2. 尽量确认问题属于 g2rain 本身，而不是环境或依赖配置问题。

创建 Issue 时建议包含：

- 使用的仓库名称与版本（例如：`g2rain-main-shell@1.0.0`）。
- 运行环境（Node 版本、浏览器、操作系统等）。
- 复现步骤（最好附上最小可复现示例）。
- 实际行为 vs 期望行为。

---

## 3. 提交 Pull Request（PR）流程

1. **Fork 仓库**  
   在 GitHub 上 Fork 目标仓库（例如 `g2rain-main-shell`），并 Clone 到本地。

2. **创建分支**  
   建议从 `main` 或对应的开发分支拉一个新分支：

   ```bash
   git checkout -b feat/xxx
   # 或
   git checkout -b fix/issue-123
   ```

3. **安装依赖 & 本地运行**  
   按各子仓库的 `README.md` 中的说明进行：

   ```bash
   npm install
   npm run dev
   # 或
   npm run build
   ```

4. **保持提交粒度清晰**  
   - 一个 PR 尽量专注于一个问题或一个特性。
   - Commit 信息建议遵循简单的约定式（如 `fix: xxx`、`feat: yyy`、`docs: update zzz`）。

5. **编写或更新测试/示例（如果适用）**  
   - 为新功能添加必要的测试或示例。
   - 确保现有测试都能通过。

6. **提交 PR**  
   - 填写清晰的标题和描述。
   - 链接相关 Issue（例如：`Close #123`）。
   - 说明改动动机、设计思路以及可能的影响范围。

7. **Review 与修改**  
   Maintainer 或 Committer 会进行 Review，并可能提出修改建议。  
   请根据反馈更新 PR，直到达成共识。

---

## 4. 代码风格与规范

不同子项目可能有各自的 Lint / 格式化配置（如 ESLint、Prettier 等），一般建议：

- 保持与现有代码风格一致。
- 提交前尽量执行：

  ```bash
  npm run lint
  npm run test   # 如有
  ```

- 避免在无必要时大规模重排代码，以免影响 Review。

---

## 5. 文档贡献建议

文档同样重要，甚至比代码更容易帮助新用户上手。你可以：

- 为新特性补充文档说明。
- 将零散的 Issue / 评论中的经验整理成正式文档。
- 补充中文/英文双语文档（如果社区有此需求）。

更新文档时建议：

- 保证示例可运行、配置项与实际代码同步。
- 在需要时添加简要的架构图或时序图，帮助理解。

---

## 6. 提案（RFC）与大改动

对于架构级别变更或影响较大的新特性，建议先通过 RFC 讨论：

1. 在本仓库或相关子仓库创建 Issue，简单描述你的想法。
2. 在 `docs/rfcs/` 下新增一个 RFC 草案，详细阐述动机、设计、方案对比等。
3. 通过 Discussions 或社区会议收集反馈。
4. 待 Maintainer 达成共识后再开始实现。

这样可以避免重复劳动，也有助于社区理解设计背景。

---

## 7. 行为准则

所有参与者都应遵守项目的 [行为准则](CODE_OF_CONDUCT.md)，保持友好、专业、尊重的交流氛围。  
违反行为准则的行为可能导致 Issue / PR 被关闭，严重情况会被限制参与。

---

## 8. 致谢

感谢每一位使用、传播和贡献 g2rain 的开发者。  
你的任何一点改动（哪怕只是修正一个错别字）都在帮助这个项目变得更好。
